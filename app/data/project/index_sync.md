# 索引同步机制说明

一诺问答并不是每次启动都盲目全量重建所有文档，而是实现了一个相对简单但足够清晰的索引同步机制。它的目标不是做数据库级别的复杂版本控制，而是确保：当文档新增、修改或删除时，本地 chunk 索引能和真实文档保持一致。

## 一、涉及的两个核心文件

当前索引同步依赖两个本地文件：

1. `doc_registry.json`
   用于记录文档级状态。每个文档会有一条记录，至少包括：
   - `doc_id`
   - `source`
   - `content_hash`
   - `chunk_count`

2. `chunks.jsonl`
   用于记录 chunk 级文本数据。每一行是一个 chunk 的 JSON 对象，至少包括：
   - `doc_id`
   - `source`
   - `chunk_id`
   - `text`

## 二、为什么要引入 doc_id

`doc_id` 是文档在系统内部的主键。  
当前推荐做法是使用相对于数据目录的相对路径，例如：

- `project/overview.md`
- `project/api_spec.md`
- `business/role_policy.md`

这样做有两个好处：

1. 比单纯文件名更稳定，也能避免重名冲突
2. 不依赖绝对路径，便于项目迁移和展示

## 三、为什么使用 content_hash

系统当前以文档全文的哈希值作为“内容是否变化”的主要依据。  
之所以不单纯依赖文件修改时间，是因为：

- 修改时间可能变化，但内容未必真的变化
- 复制、移动、编辑器保存等操作可能引起时间戳变化
- 全文 hash 更直接反映内容本身

当前推荐使用 SHA-256 对全文进行哈希。

## 四、同步逻辑概述

同步流程可以概括为：

1. 扫描数据目录中的文档
2. 读取旧的 `doc_registry.json`
3. 读取旧的 `chunks.jsonl`
4. 对当前每份文档计算 `content_hash`
5. 根据 hash 对比决定：
   - 新文档：新建 chunk 和 registry
   - 未变文档：复用旧 chunk 和旧 registry
   - 已变文档：重建该文档对应的 chunk，并更新 registry
   - 已删除文档：自动从新的结果中消失

## 五、一个重要实现原则

当前版本中，同步过程更推荐采用：

- 构造 `new_registry`
- 构造 `new_chunks`

而不是在旧集合上原地删删改改。  
这样做的原因是：

1. 逻辑更稳
2. 不容易出现一边遍历一边删除的问题
3. 更适合当前文档规模不大的项目
4. 更方便调试和验证结果

## 六、为什么 index 目录本身不能被再次索引

`app/data/index/` 目录中的 `doc_registry.json` 和 `chunks.jsonl` 是运行时索引文件，不应该再被 `loader` 当作文档读回系统。否则会出现“索引文件进入知识库”的错误递归。

因此，在扫描文档目录时，系统会跳过 `index` 目录。

## 七、当前版本的局限

尽管索引同步机制已经足够完成演示与训练目标，但它仍有局限：

- embedding 尚未持久化，服务启动时仍需重新做向量化
- 当前适用于 `txt / md`，尚未支持 PDF 等复杂格式
- chunk 级更新仍较粗糙，没有做到真正细粒度差分
- registry 和 chunks 仍是本地文件，不是数据库

不过，对当前项目而言，这套机制已经足以体现“增量索引”和“本地工程化”的思路。

## 八、调试建议

如果发现索引行为异常，可以优先检查：

- `doc_registry.json` 是否为空或损坏
- `chunks.jsonl` 是否存在不合法 JSON 行
- `doc_id` 是否和数据目录结构一致
- 新增文档是否真的被 `loader` 扫描到
- index 目录是否被误读入知识库

这几个点是当前最常见的同步问题来源。
