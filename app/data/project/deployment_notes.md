# 部署与运行说明

当前项目定位为本地运行的知识问答服务，因此部署方式相对简单，但仍有一些必须明确的约定。

## 一、模型服务

本项目当前使用 Ollama 作为本地模型服务，默认模型为：

- `qwen3:8b`

模型调用通过本地兼容接口完成，默认地址为：

- `http://localhost:11434/v1/`

当前版本不依赖云端 OpenAI API key，也不需要外部网络推理服务。只要本地 Ollama 能正常运行，并且 `qwen3:8b` 已经下载完成，系统就能进行本地问答。

## 二、服务启动

当前推荐使用：

```bash
python -m uvicorn app.main:app
在本项目中，不推荐在某些阶段长期使用 --reload。原因是项目启动时会执行索引同步，而索引同步会写入 app/data/index/ 下的文件。如果服务同时监控这些文件变化，就可能出现反复重载，影响 /docs 页面稳定性。

三、为什么 docs 有时难以打开

如果使用 --reload 并且启动时写入索引文件，可能出现以下现象：

服务重复重载
/docs 页面刷新不稳定
终端中不断出现重载相关日志

因此，当前阶段更推荐在功能验证时先关闭自动重载，待后续把索引目录从监控中排除后，再恢复更方便的开发模式。

四、项目启动流程

当前版本的大致启动顺序如下：

构建 pipeline
在 pipeline 构建过程中调用索引同步
读取 chunk 数据
对 chunks 做 embedding
构建向量库
初始化 retriever、reranker、llm_client
启动 FastAPI 服务

这种做法的缺点是：首次启动比普通 API 服务更慢。
优点是：一旦服务起来，后续 /ask 调用链路会更完整。

五、当前运行假设

为了保证系统能正常启动，当前默认有以下假设：

app/data/ 目录存在
app/data/index/ 可写
Ollama 已安装并能正常运行
qwen3:8b 已经可以在终端通过 ollama run qwen3:8b 唤起
Python 虚拟环境中的依赖已安装完成
六、常见问题
1. 端口被占用

如果 8000 端口被旧的 Python/Uvicorn 进程占用，服务将无法正常启动。
这种情况下通常需要查找占用端口的进程并结束之。

2. 空的 registry 文件

如果 doc_registry.json 文件存在但为空，直接 json.load 会失败。
因此读取逻辑应对空文件做保护。

3. chunks.jsonl 中混入坏数据

如果 chunks.jsonl 某一行是 {} 或其他不合法结构，Chunk(**item) 会抛校验错误。
因此读取时应跳过空对象或损坏行。

七、后续可优化方向

后续部署层面的优化方向包括：

为索引目录增加 reload exclude 配置
embedding 持久化，减少启动成本
Docker 化部署
提供简单的健康检查接口
提供“只重建索引，不重启服务”的能力
