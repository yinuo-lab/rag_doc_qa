# 系统架构说明

一诺问答采用相对清晰的模块化设计。项目虽然规模不大，但刻意将文档加载、切分、检索、重排、Prompt 构造、模型调用和流程编排分开，以便后续替换局部实现，而不需要推翻整套系统。

当前主要模块如下：

1. `loader.py`
   负责扫描文档目录，读取支持的文件类型，并构造 `Document` 对象。Document 至少包含三个字段：`doc_id`、`source`、`text`。其中 `doc_id` 是系统内部主键，建议使用相对于数据目录的相对路径；`source` 是给用户看的来源名称，通常是文件名；`text` 是文档全文。

2. `chunker.py`
   负责把全文切成多个 chunks。每个 `Chunk` 至少包含 `doc_id`、`source`、`chunk_id`、`text`。`chunk_id` 表示该 chunk 在所属文档中的顺序编号。切分时允许 overlap，目的是尽量减少语义被硬切断带来的信息丢失。

3. `embedder.py`
   负责把 query 或 chunk 文本转成向量。当前版本使用的是最小化示例实现，不是生产级 embedding 模型。这个模块的定位更像“结构占位”，便于后续替换成更真实的 embedding 服务。

4. `vector_store.py`
   负责保存 chunk 和 embedding，并提供相似度搜索能力。当前版本是内存型向量存储，主要用于验证流程。它不是持久化数据库，也不是高性能 ANN 检索系统。

5. `retriever.py`
   负责 query 向量化和候选召回。它不关心模型生成，也不负责 Prompt 组织。其职责是“给后面模块一个相对合理的候选集合”。

6. `reranker.py`
   负责在召回结果之上做更细粒度的排序。当前版本使用较简单的重排逻辑，重点在于搭起“召回 + 重排”的结构，而不是追求最强排序性能。

7. `prompt_builder.py`
   负责把用户问题和检索到的资料拼成模型输入。系统强调：Prompt 必须明确告诉模型“优先基于资料回答；如果资料不足，应该明确说不知道”。这个模块对于减少幻觉非常重要。

8. `llm.py`
   负责与本地模型交互。当前通过 Ollama 本地服务调用 `qwen3:8b`。项目并不直接依赖云端 OpenAI API，而是走本地兼容接口。

9. `rag_pipeline.py`
   负责把检索、重排、Prompt 构造和模型调用串成一个完整流程。它是整个问答系统的核心编排层，但不应该在这里堆太多底层细节。

10. `index_store.py`
    负责本地索引状态管理，包括 `doc_registry.json` 与 `chunks.jsonl` 的读写，以及文档同步逻辑。这个模块是项目工程化的关键点之一。

从职责边界上看，系统刻意避免让路由层直接操作检索细节，也避免让检索模块负责生成答案。这样做的原因是：当项目从演示型 demo 逐步走向更像样的服务时，清晰的边界能显著降低后续重构成本。

项目的接口层由 FastAPI 提供。当前只暴露了 `/ask` 路由，但未来可以继续扩展 `/health`、`/reload-index`、`/debug/retrieval` 等接口。架构上建议坚持一个原则：接口层负责接收请求和返回响应，业务链路放在 pipeline 或 service 层。

在启动阶段，系统会先同步索引，再构建向量库。之所以不把所有逻辑都塞进路由，是因为索引同步和 embedding 初始化都更适合在服务启动时统一处理，而不是每次请求都重复执行。当前版本中，索引更新和向量构建发生在服务初始化阶段，这使得首次启动比普通 FastAPI 服务更慢，但后续问答响应会更稳定。

总体来说，这个架构的重点不在“复杂”，而在“边界明确、可以替换、能讲清楚”。这对于工程训练和面试表达都很重要。
