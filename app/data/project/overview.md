# 项目总览：一诺问答（Yinuo QA）

一诺问答是一个面向本地知识文档的检索增强生成系统，用于验证“本地文档 + 本地模型 + API 服务化”这一组合在实际工程中的可行性。项目当前定位不是通用聊天机器人，而是一个更偏“私有文档问答”和“内部知识查询”的实验性服务。

本项目的目标分为三层。第一层是基础能力，也就是把文档读进系统、切成多个 chunks、建立索引、根据问题检索相关内容，再把这些内容交给本地模型生成答案。第二层是工程能力，包括模块分层、索引同步、错误处理、来源返回和启动流程管理。第三层是展示能力，也就是让整个项目具备较好的可运行性、可解释性和可演示性，能够通过 `/ask` 接口统一对外提供问答服务。

当前版本中，项目的核心输入是本地 `txt` 和 `md` 文档，核心输出是结构化问答结果，其中响应体包含 `answer` 和 `sources` 两部分。`answer` 用于表达最终回答，`sources` 用于告诉用户回答主要参考了哪些文档片段。系统强调“来源可追踪”，因为在知识问答场景下，仅仅给出一个流畅答案并不够，用户通常还希望知道这个答案来自哪份材料。

一诺问答默认采用本地模型推理。当前使用的本地模型是 `qwen3:8b`，通过 Ollama 提供服务接口。项目之所以选择本地模型，而不是直接依赖云端 API，主要是出于三点考虑：第一，减少外部调用成本；第二，降低对网络与外部平台的依赖；第三，便于后续测试“本地知识 + 本地推理 + 本地服务”的完整链路。

从架构角度看，本项目的主流程可以概括为：

Document -> Chunk -> Retrieval -> Rerank -> Prompt -> LLM -> Answer

其中：
- Document 表示原始文档；
- Chunk 表示切分后的文本片段；
- Retrieval 表示检索候选片段；
- Rerank 表示对候选结果进行重排；
- Prompt 表示将问题和资料拼接成模型输入；
- LLM 表示本地大模型；
- Answer 表示最终返回给用户的答案。

项目还实现了本地索引同步逻辑。系统会扫描文档目录，对每个文档计算内容哈希，并使用 `doc_registry.json` 记录文档级状态，使用 `chunks.jsonl` 记录 chunk 级数据。如果文档内容没有变化，则尽量复用旧的索引内容；如果文档新增、修改或删除，则更新本地索引。这样做的目的是减少重复工作，并让系统具备基本的工程维护能力。

需要注意的是，一诺问答不是为了证明“大模型懂不懂 RAG 常识”，而是为了验证系统能否从给定文档中提取模型原本不知道、或者不应该凭常识乱答的信息。因此，测试集会刻意包含项目内部规则、接口约定、部署说明、角色权限和干扰文档，以检验检索质量和来源控制能力。

当前版本代号为 **Yinuo-QA v0.3**。在这一版本中，系统已经具备基础可运行能力，但仍有若干待优化点，例如 embedding 持久化、相关性阈值控制、多格式文档支持和更强的 reranking 机制。
