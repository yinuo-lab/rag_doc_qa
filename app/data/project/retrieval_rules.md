# 检索与重排规则

一诺问答当前采用“召回 + 重排”的两阶段策略。虽然底层实现仍偏演示性质，但系统已经明确区分了“先尽量找全”和“再尽量排准”这两个目标。

## 一、召回阶段

召回阶段的目标是：尽量不要漏掉可能有用的 chunks。当前默认配置如下：

- `retriever top_k = 8`
- 每次检索先从全部 chunk 中取出前 8 个候选结果
- 相似度度量采用向量相似度，演示版本中主要使用 cosine similarity 思路

为什么默认不是 `top_k = 3`？  
因为如果一开始就取太少，很容易在召回阶段把本来相关但排序稍靠后的 chunk 丢掉。当前策略是先略宽松地召回，再在第二阶段做重排。

## 二、重排阶段

重排阶段的目标是：在候选结果中找出更可信、更接近用户问题的部分。当前默认配置如下：

- `reranker top_n = 3`
- 也就是说，重排后最多保留 3 个主要候选

当前版本的重排逻辑偏简单，更多是工程占位，不是追求最高检索指标。后续如果替换成 cross-encoder 或更强的 reranker，系统结构不需要大改。

## 三、来源筛选

为了让最终返回给用户的 `sources` 更干净，系统还会做额外筛选。当前约定如下：

1. 只有 `score > 0` 的结果才算“有效来源”
2. 同一个 `doc_id` 只保留一个 source，避免同一份文档反复出现
3. 如果多个 chunk 都来自同一文档，优先保留得分更高或排序更靠前的那一个
4. 如果最终没有有效来源，模型应尽量回答“不知道”，而不是硬编

这套规则的意义是：回答可以适当综合，但来源展示要尽量简洁，不要把一堆弱相关文档全带上。

## 四、无答案策略

当问题和文档无关时，系统不应强行输出貌似正确的答案。  
例如，当用户提问“谁是美国总统”这类与当前知识库无关的问题时，系统理想行为是：

- 明确表示资料不足
- 不要依赖模型通识硬答
- 尽量返回“不知道。提供的资料中没有足够信息。”

## 五、相关性控制原则

当前版本更关注“结构正确”和“来源可解释”，而不是追求特别复杂的排序算法。因此，检索与重排遵循以下原则：

- 优先可解释
- 优先工程可控
- 优先减少明显无关来源
- 允许后续再通过更强 embedding 或 reranker 提升效果

## 六、为什么不能只测公开常识

如果知识库里全是“RAG 是什么”“FastAPI 是什么”这类公开常识，大模型可能本来就知道，即使检索没做好，也可能答得像对的一样。因此，项目测试更强调：

- 私有配置
- 内部规则
- 人工构造的业务约束
- 需要跨文档拼接的信息

这才更能检验 RAG 系统本身的价值。

## 七、当前约束总结

为了便于测试，当前版本约定的关键检索参数如下：

- 文档切分：默认 `chunk_size = 220`
- 文档切分：默认 `overlap = 60`
- 召回：`top_k = 8`
- 重排：`top_n = 3`
- 来源过滤：仅保留 `score > 0`
- 来源去重：按 `doc_id` 去重

后续如果参数调整，建议优先修改文档并同步更新测试用例。
